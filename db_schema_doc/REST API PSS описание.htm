Особенности использования REST API в сервере A-YATZK

Сервер предоставляет базовое REST API, которое позволяет оперировать объектами низкого уровня – инстансами, их атрибутами, БЛОБами. В сервере A-YATZK базовое REST API можно дополнить путем написания плагинов к серверу приложений; для серверов PSS механизм плагинов отсутствует. Плагины задаются в файле настроек в группе [Plugins] параметрами rest=*, см Расширения REST API в плагинах сервера приложений.

Все команды базового REST API начинаются с элемента /rest/. Для плагинов необходимо использовать другой стартовый элемент, но при необходимости можно перегружать отдельные операции базового API.

Сервер A-YATZK (только сервер A-YATZK) может предоставлять базовые функции веб-сервера. Для работы в этом режиме, в файле настроек сервера необходимо создать описание одного или нескольких сайтов. Для каждого сайта необходимо указать каталог для статических файлов; при необходимости можно указать способы аутентификации, БД для выполнения авторизации и т.п. Сайты задаются в файле настроек в группе [Plugins] параметрами http=*. Подробное описание настроек сайтов см Настройка сайтов веб-сервера..

Перед чтением и изменением данных через REST API необходимо выполнить аутентификацию и авторизацию в БД. Для этого можно использовать два метода:

1) собственный механизм аутентификации и авторизации. Сессия начинается командой /rest/connect и завершается командой rest/disconnect. В процессе авторизации требуется передать на сервер логин и пароль пользователя, а также, название БД. Список доступных БД можно узнать командой /rest/dblist. Этот механизм работает и в A-YATZK, и в PSS.

2) стандартные способы аутентификации веб-приложений HTTP Basic или HTTP Negotiate + ALD. Эти способы работают только при обращении к настроенным сайтам, у которых указан как минимум один способ аутентификации, и задана БД для авторизации пользователей. Этот механизм работает только в A-YATZK.

Выбор способа аутентификации зависит от задач конкретной системы. Так, аутентификация собственным механизмом позволяет сделать гибкую систему, позволяющую начинать и завершать сессии и работать с разными БД. В то же время использование веб-аутентификации позволяет сделать систему сквозной аутентификации.

По итогам авторизации сервер формирует код сессии, который возвращает в json и записывает в куки. Для использования авторизованной сессии можно передавать этот код в специальном http-заголовке (A-YATZK|PSS), или можно использовать стандартную обработку куков браузером (только A-YATZK).

Расширения REST API в плагинах сервера приложений.

Для создания собственных вариантов REST API можно использовать механизм плагинов сервера приложений (только A-YATZK).

Плагины к серверу, это консольные приложения, разработанные на основе библиотеки AplSrvPluginCore. Плагин должен быть задан в файле настроек AplTransport.ini сервера приложений в секции [Plugins]; каждая строчка задает один плагин. Описание одного плагина имеет следующий вид:

rest=<A>|<B>[|<C>|...]

Где rest это тип плагина, A, B и т.д. это параметры плагина:

<A> – первый элемент rest-запроса, обработка которого должна передаваться в плагин. Например, для обработки плагином запросов вида http://<server>/docapi/DDD/BBB/YYY в параметре A должна быть строчка «docapi».

<B> – путь к исполняемому файлу плагина. Можно указывать абсолютный путь, можно указывать относительный путь. Путь определяется относительно файла настроек ini.

<C> – режим работы плагина. Используется для работы одного плагина в разных режимах. Эта строчка передается плагину при его инициализации и доступна из программного кода.

При необходимости обработки плагином запросов, начинающихся с /rest/, можно использовать следующую форму записи параметра <A>: rest/<второй элемент запроса>,<еще один второй элемент запроса>… .
Например, если плагин должен обрабатывать запросы  /rest/get_role/* и /rest/messages/* то запись может иметь следующий вид:

rest= rest/get_role,messages|.\aplSrvPluginRbt.exe
 

Примеры задания плагинов:
[Plugins]

rest=test-tanki|v:\__QT\build-A-YATZK-Debug\_StepData\aplSrvPluginDemo \debug\aplSrvPluginDemo.exe|tanki

rest=test-avia|aplSrvPluginDemo.exe|avia

Подробное описание разработки и работы плагинов см документ «REST API Плагины сервера приложений для обработки HTTP.docx»  

Настройка сайтов веб-сервера.

Сервер A-YATZK может работать в режиме веб-сервера. Для этого необходимо создать описание одного или нескольких сайтов. Этот режим может сочетаться с обычным режимом работы с бинарными клиентами или обслуживания запросов REST API.

Поддерживаются следующие фичи:
- выдача статических файлов
- аутентификация HTTP Basic или HTTP Negotiate (на 2024 год аутентификация Negotiate только в Linux + домен ALD)
- настраиваемые редиректы (перенаправления)
- настраиваемые CORS

Параметры сайтов задаются в AplTransport.ini в секции [Plugins] параметрами http. Вид строки с описанием сайта имеет следующий вид:

http=<A>|<B>[|<C>|...]

Где A, B и т.д. это параметры сайта:

<A>: доменное имя сайта. Имя должно быть указано с портом, например server.domain:7239

<B>: путь к каталогу с файлами сайта. Можно указывать абсолютный путь, можно указывать относительный путь. Путь определяется относительно файла настроек ini.

<C>: маска индексного файла. Если не задана, используется маска index.htm*

<D>: параметры аутентификации. Состоят из строчки следующего формата:

              d:<Имя_бд>,b<+|->,n<+|->,s:<SPN>,k:<KEYTAB>

              где: <Имя_бд> имя БД, с которой происходит работа при rest запросах этого сайта и из которой берутся данные пользователей для аутентификации и авторизации. Обязательный параметр. БД должна быть задана в списке Lite БД для Lite сервера и в списке БД для PostgreSQL  в случае сервера для PostgreSQL.

b - параметр аутентификации basic. Если аутентификация basic нужна, после b ставится '+'; если не нужна - '-'

              n - параметр аутентификации negotiate (kerberos). Если аутентификация negotiate (kerberos) нужна, после k ставится '+'; если не нужна - '-'.

              <SPN> - SPN службы сервера. Должен быть задан, если включена аутентификация negotiate (kerberos)

              <KEYTAB> - путь к файлу keytab. Должен быть задан, если включена аутентификация negotiate (kerberos)(только под linux)

 

 

Примеры записи сайта (первый для windows, второй и третий для linux):

http=localhost:7239|d:\_swh_www\ |index.htm*|d:c_test,b+,n-

 

http=ald103srv.ald103.test:7239|/opt/www/site1/|index.htm*|d:c_test,b+,n+
,s:HTTP/ald103srv.ald103.test@ALD103.TEST,k:/opt/keytab/HTTP_ald103srv.keytab
http=ald103srv.ald103.test:7241|/opt/www/site2/|index.htm*|d:PG_test,b+,n+
,s:HTTP/ald103srv.ald103.test@ALD103.TEST,k:/opt/keytab/HTTP_ald103srv.keytab

Правила перенаправления (редиректы)

Редиректы задаются в группе [HttpRedirect]. На каждой строчке задается один редирект. Каждый редирект имеет вид <правило>=<значение>.

Если правило начинается с '/', то оно работает для всей строки запроса, начиная с корня. Иначе сравнивается только имя файла после последнего '/'.

В правиле может присутствовать символ подстановки '*' - он обозначает любое количество символов. Пока что он может находиться только справа или только слева строки правила. Например, под правило '*.js' подпадут все файлы js, а под правило '.*' все файлы, начинающиеся с точки.

Значение определяет, куда пойдет редирект и пойдет ли вообще. Если значение равно пустой строке, сервер формирует ответ 404. Если в значении указан адрес, формируется ответ 301 с перенаправлением на этот адрес. Например, правило
/upload-packages*=/
перенаправит запросы /upload-packages123, /upload-packages/abc и подобные на корень сайта.

Значения могут быть только абсолютными, подстановок из текущего адреса в адрес редиректа пока что нет.

Настройка CORS

CORS задается в группе [Options] параметром permittedCrossRequest. После «равно» идет перечисление адресов, с которых разрешены междоменные запросы. Адреса разделяются пробелами. Звездочка обозначает разрешение запросов с любых адресов.

Пример:

permittedCrossRequest=http://192.168.15.68:5173 *

Чтение списка БД

Формат вызова:

GET /rest/dblist/

При успешном выполнении запроса сервер возвращает http код 200 и JSON со списком БД в таком формате:

{"dblist": [

<имя БД 1>,

… ,

<имя БД N>

]

}

Список отсортирован по строкам по возрастанию, например:

{

"dblist": [

"LSA-SERVER",

"SWH11",

"SWH_115",

"VS-TEST2"

]

}

Запрос версии сервера

Формат вызова:

GET /rest/version/

При успешном выполнении запроса сервер возвращает http код 200 и JSON c версией сервера в таком формате:

{"version": <строка версии>}

Сервер A-YATZK возвращает в значении "строка версии" содержимое файла version.txt, находящегося в каталоге с сервером. Сервер PSS возвращает значение версии из файла *.ver, находящегося в каталоге с сервером. Если файла нет, оба сервера возвращает дату и время сборки; сервер возвращает имя исполняемого файла сервера.

 

Соединение с БД

Все параметры передаются в командной строке; пароль не шифруется. Формат вызова:

GET /rest/connect/user=<логин>&pass=<пароль>&db=<имя бд>&lic=<тип лицензии>

Все параметры – строки. Регистр символов важен. Если какая-то строка пустая, например пустой пароль, то можно параметр не указывать, или указывать в виде user=tester&pass=&db=test_db&lic=ILS

При необходимости передать в логине/пароле/имени БД пробелы/кавычки/русские буквы/прочие спецсимволы они кодируются стандартным образом в виде %20 (или для русских символов – в UTF8 варианте; таблица кодирования, например, вот тут: http://web-developer.name/urlcode/).

Второй вариант на будущее – параметры передаются в теле запроса, пароль кодируется с помощью стандартного механизма PSS и т.п.

Возвращаемое значение

При успешном подключении сервер возвращает http код 200 и JSON с кодом сессии и дополнительными данными в таком формате:

{"session_key": <строка с кодом сессии>,

"error_description": <строка с описанием ошибки>,

"_serverVersion": <версия сервера приложений>,

"_serverVerDict": <версия словаря БД>

}

Платформа A-YATZK дополнительно возвращает имя БД, и присоединенного пользователя, описание приложения, а также время запроса и время выполнения запроса.

Если соединение проходит по логину-паролю, но есть предупреждения (у пользователя режим "только чтение", заканчивается срок лицензионного ключа и т.п.) то в выходной буфер добавляется http-заголовок с именем «X-APL-Warning» равный 1, а описание ошибки помещается в корневой блок JSON в параметр error_description. В случае отсутствия предупреждений параметр error_description не заполняется.
Параметры "_serverVersion" и "_serverVerDict" могут пригодиться при проверках совместимости приложения с сервером.

Код сессии так же прописывается в куке X-Apl-SessionKey; поэтому при следующих запросах код сессии будет автоматически прочитан браузером из куков и отправлен на сервер. Если куки по какой-то причине не работают (например, отладка из Postman), то необходимо отправлять код сессии в виде http заголовка с именем «X-Apl-SessionKey». Код представляет набор латинских букв и цифр, поэтому дополнительного кодирования при передаче не требуется.

Для отладочных целей можно передавать код версии так же в строке URI в блоке rest:

GET /rest&session_key=<строка с кодом сессии>&…/.../…/

При одновременном указании кода сессии в http-заголовке и в виде параметра команды приоритет у кода параметра команды rest из строки URI.

Например, соединяемся с БД testdb под пользователем Test0 и паролем 1234:

http://localhost:7239/rest/connect/user=Test0&pass=1234&db=testdb/

Сервер может вернуть такой ответ с заголовком X-APL-Warning равным 1 и таким текстом:

{

"_serverPort": "7239",

"_serverVersion": "5.231_ru",

"_serverVerDict": "1225",

"session_key":"FDPGJCOFIALFDFGBNEDBAHBCAGHDCEEEAAGGACEGJCKHNHIGOCJEDAAFHENAKEMAHBJHNDKHEDGEPBPD",

"error_description": [

        "Ваш доступ к БД ограничен чтением."

    ]

}

Если параметры авторизации указаны синтаксически корректно, но не позволяют выполнить авторизацию, то сервер возвращает http код 403. Если параметры авторизации заданы некорректно, например, нет обязательного параметра – сервер возвращает код 400. Ну и если сервер не может соединиться с БД или если БД заблокирована администратором – возвращается код 503. В случае любых ошибок в корневом блоке JSON в параметре error_description указана поясняющая информация. Например, при указании пароля в неправильном параметре passw сервер вернет http код 403 и такое сообщение в теле ответа:

{

"error_description":

[

"Неверное имя пользователя или пароль",

"В блоке параметров \"user=Test0&passw=1234&db=testdb\" найден неподдерживаемый параметр passw. Допустимые параметры для блока: user, pass, db, lic"

 ]

}

Завершение сессии

Формат вызова:

GET /rest/disconnect/<код сессии>

Код завершаемой сессии передается или в куке X-Apl-SessionKey, или в http заголовке «X-APL-SessionKey», или параметром команды rest:

POST /rest&session_key=<код сессии>/disconnect/

Возвращаемое значение

Сервер возвращает ответ 204, ответные данные – JSON со служебной информацией о времени выполнения команды. Если такая сессия не найдена – ситуация не обрабатывается, клиенту так же возвращается 204.

 

Запрос имени текущей БД

Только сервер A-YATZK.

Формат запроса:

GET /rest/get_current_db 

Параметров нет.

В ответ сервер возвращает JSON-буфер. Результат имеет следующий формат:

{

curr_db: string – имя текущей БД

}

 

Запрос имени текущего пользователя (базовый вариант)

Только сервер A-YATZK.

Формат запроса:

GET /rest/get_current_user 

Параметров нет.

В ответ сервер возвращает JSON-буфер. Ответ сервера A-YATZK имеет следующий формат:

{

id: # int – идентификатор инстанса текущего пользователя

curr_user: string – логин текущего пользователя

fullname: string, - полное имя. Формат строки: «Описание (логин)» (если у пользователя задано описание) или «Логин» (если описания нет)

description: string – описание текущего пользователя

}

 

 

Сортировка и разбиение результатов запроса на порции

Для запросов чтения данных load, find и query можно задать параметры сортировки и разбиения по порциям. Эти параметры указываются в первом блоке после кодового слова rest и отделяются от него амперсандами:

GET /rest&sort=<имя атрибута>&start=<начальный индекс>&size=<размер порции>/.../…/

Описание параметров:

·       sort – имя атрибута, по которому будет отсортирован выходной массив инстансов. Можно указывать как имя атрибута, так и число – идентификатор атрибута. Под сортировку попадают все инстансы, имеющие указанный атрибут, в т.ч. если указано имя атрибута – то разных типов. Например, если два подзапроса возвращают документы и изделия, и задана сортировка по атрибуту «id», то документы и изделия будут перемешаны и отсортированы по строчкам в этом атрибуте. Остальные инстансы включаются в список после отсортированных по атрибуту и отсортированы по идентификаторам инстансов. Если атрибут не указан – весь выходной экстент отсортирован по идентификаторам инстансов. По умолчанию сортировка выполняется по возрастанию.

·       sort_desc – если задан параметр, то сортировка идет по убыванию.

·       start – начальный индекс в массиве. Индексы начинаются с 0.

·       size – размер порции.

·       no_filter_spec – (только сервер проекта Ц) отключить фильтрацию по специальным атрибутам.

Если размер порции не задан, используется размер по умолчанию

Если атрибут для сортировки не задан, массив будет отсортирован по идентификатору инстанса; сортировка всегда по возрастанию. В настоящее время сортировка выполняется только по атрибутам типа "строка" (aplSTRING); инстансы, у которых атрибут с указанным именем является строковым, попадают в первую часть выходного массива и отсортированы по указанному атрибуту. Остальные инстансы добавляются во вторую часть выходного массива и отсортированы по идентификатору инстанса.

Если у нескольких инстансов совпадают значения атрибута, указанного для сортировки, то такие инстансы помещаются в выходной буфер в порядке возрастания идентификаторов инстансов.

На порции согласно параметрам start и size массив делится после сортировки и объединения двух частей выходного массива в один массив.

Если размер порции не задан, используется размер по умолчанию (100). Если заданный размер порции превышает максимально возможный размер (500), то размер порции уменьшается до максимально возможного размера.

Особенности запросов load

·       под сортировку подпадают только инстансы первого уровня вложенности, т.е. результаты запросов 'a' и 'e'. Инстансы второго и т.д. уровней в общий массив не попадают; инстансы, полученные в результате обратных подзапросов ('b' и 'r'), добавляются в конец списка и сортируются по идентификатору инстанса.

·       если какая-либо инстанс будет получена и запросом первого порядка, и каким-либо последующим обратным запросом, то она будет фигурировать только в первой части списка (в части с сортировкой по заданному атрибуту). Аналогично, если инстанс будет получена в запросах первого порядка несколькими способами (например, в запросе 'a' по конкретному идентификатору и в запросе 'e' по типу), то в списке она будет присутствовать в единственном числе.

·       если при запросе load в качестве атрибута для сортировки указан атрибут, который не читается в массиве подзапросов (например, есть подзапрос типа 'e' с параметром all_attrs= false и сортировкой по атрибуту этой ентити), то порядок таких инстансов в выходном буфере непредсказуем.

Обработка некорректных значений

Неправильное указание параметров сортировки не меняет код возврата; при корректнгых параметрах запроса сервер возвращает код возврата 200. В то же время при некорректных значениях http-заголовок X-APL-Warning устанавливается в значение 1, а в выходной буфер JSON в массив строк error_description добавляется строка с описанием ошибки.

В случае некорректного указания имени атрибута или несоответствия его типа выполняется сортировку по идентификатору инстанса; в массив строк error_description добавляется строка с описанием ошибки.

Если указан отрицательный параметр start, то он меняется на 0 и возвращаются инстансы начиная с нулевого. Если указан параметр start больший, чем количество найденных инстансов, то инстансы не возвращаются, а в поле count_all возвращается реальное количество найденных инстансов (см. Представление инстансов и атрибутов).

Если указан отрицательный параметр size, то он меняется на размер по умолчанию (100). Если указан параметр size больший, чем максимально допустимый (500), то размер порции меняется на максимально допустимый.

 

Запрос данных традиционного формата

Запрос одиночного инстанса со всеми атрибутами:

GET /rest/i=<число - идентификатор инстанса>

Запрос списка инстансов определенной энтити:

GET /rest/ent=<строка – имя энтити>

Можно задать атрибут, по которому будет отсортирован список инстансов:

GET /rest/ent=<строка – имя энтити >&sort=<имя атрибута>

Если атрибут не задан, массив будет отсортирован по идентификатору инстанса; сортировка всегда по возрастанию.

 При запросе инстанса можно опускать ключ i. При запросе списка инстансов ключ ent указывать обязательно, так как в будущем нельзя исключить пересечение имен энтитей и кодовых слов для сложных запросов.

Возвращаемое значение

При успехе запроса данных сервер возвращает код http 200, а в теле – JSON-буфер универсального формата (см. Представление инстансов и атрибутов).

Если запрос выполнен на неавторизованной сессии (не указан код сессии или сессия с указанным кодом не существует) то сервер возвращает код 401 и добавляет в выходной JSON в массив error_description строку с описанием ошибки:

{"error_description": ["Указанный код сессии не действителен. Необходимо пройти авторизацию с помощью команды /rest/connect/"]}

Если неверно указан идентификатор инстанса или имя ентити, то сервер возвращает код 404 и буфер с описанием ошибки. Если имя энтити указано верно, но для нее нет инстансов, то возвращается JSON-буфер универсального формата с пустым массивом инстансов.

 

Запрос данных Load Data

Запрос является аналогом запросов CaplLoadData из сишного API и предназначен для загрузки цепочек instance и их атрибутов, связанных прямыми и/или обратными ссылками согласно схеме данных. Для каждого звена цепочки описывается один подзапрос. У каждого подзапроса могут быть такие атрибуты:

·       идентификатор подзапроса. По нему последующие подзапросы могут ссылаться на результаты предыдущих подзапросов.

·       тип запроса (см ниже)

·       в зависимости типа подзапроса идентификатор instance или номер предыдущего подзапроса

·       entity (тип instance)

·       атрибута

·       загрузить ли все аттрибуты всех полученных в результате запроса instance в кеш клиента. Если true, то в кэш клиента читаются все атрибуты объектов, полученных в результате подзапроса. 

Типы подзапросов и их параметры

'a'

добавление самого instance в список результатов (запрос инстансы). Идентификатор instance задается в поле id.   При load_all_attrs==true так же будут загружены значения всех атрибутов.

'i'

вернуть значение атрибута у известной instance (прямой запрос). В поле id задается идентификатор instance; в поле attr имя атрибута для загрузки.

'e'

вернуть список всех instance заданного типа (запрос экстента). В поле ent указывается имя entity, которая определяет тип искомых инстансов.  Обычно используется только при зачитывании словарей.

'b'

вернуть список instance, ссылающихся указанным атрибутом на заданный instance (обратный запрос по инстансу). В поле id указывается идентификатор инстанса, на который должна быть ссылка; в поле attr задается имя атрибута, в котором должна быть ссылка.

'd'

вернуть список значений аттрибута у всех instance, определенных предыдущим подзапросом (прямой запрос по предыдущему подзапросу). В поле id указывается идентификатор предыдущего подзапроса; в поле attr имя атрибута для загрузки.

'r'

вернуть список instance определенного типа, ссылающихся указанным атрибутом на все instance, определенные предыдущим запросом (обратный запрос по предыдущему запросу). В поле id указывается идентификатор предыдущего подзапроса; в поле ent указывается имя entity, которая определяет тип искомых инстансов; в поле attr задается имя атрибута, в котором должна быть ссылка.

'u'

добавить подзапрос к виртуальной группе подзапросов

 

Подробнее про запросы LoadData можно почитать в разделе CaplLoadData, SourceSafe «$/__DOC/API/APL_API/APL_API.chm». Отличия от CaplLoadData – все промежуточные результаты попадают в буфер (параметр in_result считается всегда равным true), добавлен параметр для сортировки результата.

Формат вызова цепочки запросов (параметры аналогичны используемым в CaplLoadData, но имена параметров упрощены для облегчения использования в REST):

GET /rest<управление сортировкой и порциями>/load/<запрос 0>/<запрос 1>/…/<запрос N>

Формат одного запроса (пробелы в строке:

q=< номер запроса>&t=<тип подзапроса>&id=<идентификатор инстансы или предыдущего запроса>&ent=<имя энтити>&attr=<имя атрибута>&all_attrs=<возвращать ли все значения атрибутов>

Параметры имеют следующие значения и типы:

·       q – <номер подзапроса> – число

·       t – <тип подзапроса> – латинская буква без кавычек, см описание CaplLoadData.

·       id – <идентификатор инстансы или предыдущего подзапроса> - число.

·       ent – <имя энтити> – имя типа запрашиваемой энтити, латинские буквы без кавычек. Вместо имени можно указывать числовой идентификатор энтити.

·       attr – <имя атрибута> – имя запрашиваемого атрибута, латинские буквы без кавычек. Вместо имени можно указывать числовой идентификатор атрибута.

·       all_attrs – <возвращать ли все значения атрибутов> – булевое значение словом без кавычек, true или false. По умолчанию true (поведение отличается от CaplLoadData, в котором этот флаг по умолчанию false).

Обязательно должно быть задан тип запроса и связанные с ним поля, т.е. для запроса типа «e» обязательно должно быть задано имя энтити, для запроса «a» необходимо задать идентификатор инстансы и т.п. Если не задан параметр all_attrs, то он считается равным true.  Если не задан номер запроса – он получается путем увеличения на 1 номера предыдущего запроса; нумерация начинается с 0.

Запрос данных поиском по атрибутам find

Формат запроса:

GET /rest<управление сортировкой и порциями>/find/ent=<имя энтити>&sub=< учитывать ли при сравнении строк подстроки>&case=<учитывать регистр>&mode=<режим сравнения подстрок>&all_attrs=<возвращать ли все значения атрибутов>/< значение атрибута 1>/.../< значение атрибута N>

Параметры в целом повторяют параметры функции NET_FindInstancesWithAttrValues:

·       ent – <имя энтити> – имя искомой энтити – (обязательный параметр). Вместо имени можно указывать числовой идентификатор энтити.

·       sub – учитывать при сравнении строк подстроки или нет. Булевое значение, указывается словом без кавычек, true или false. По умолчанию – true.

·       case – учитывать при сравнении строк разницу в регистре символов. Булевое значение, одним словом, без кавычек, true или false. По умолчанию – false.

·       mode – используется при поиске по подстрокам (параметр sub=true) чтобы указать, где должна находиться искомая строка. Возможные значения (без кавычек): left, mid (значение по умолчанию), right.

·       all_attrs – <возвращать ли все значения атрибутов> – булевое значение словом без кавычек, true или false. По умолчанию true.

<значение атрибута N> представляет пару: имя атрибута = значение атрибута. Если значение атрибута состоит только из английских символов – она указывается без кавычек; если в ней встречаются пробелы, русские буквы и прочие недопустимые для строки URL символы – ее надо кодировать в процентном формате. Пример: запрашиваем папку, у которой в начале наименования может встречаться строка fam15 в любом варианте символов – строчном или прописном. Папки сортируем по атрибуту name:

Сохранение данных

Формат запроса:

POST /rest/save/

Тип содержимого запроса, указываемый в поле Content-Type: «application/json».
В теле запроса должен находиться JSON-буфер универсального формата (см Представление инстансов и атрибутов). В буфере обязательно должен быть указаны следующие заголовки:

"format": "apl_json_1",

"dictionary": "apl_pss_a"

Инстансы в буфере могут быть помещены как линейным списком, так и в виде дерева аналогичного результатам запроса load. Уровень доступа инстансов можно не указывать.

Для тех инстансов, которые надо удалить, в описании инстанса поле type должно быть значение null (без кавычек).

Можно не указывать те атрибуты инстансов, значения которых не надо менять при сохранении. И наоборот, обязательно надо указывать значение атрибута, если надо очистить предыдущее значение атрибута (например, задать пустую строчку для строкового атрибута или значение null для атрибута типа инстанс).

Удаление инстансов традиционного формата

Удаление одиночного инстанса со всеми атрибутами:

DELETE /rest/<число - идентификатор удаляемого инстанса>

В запросе можно указать несколько удаляемых инстансов через запятую, например:

DELETE /rest/4005,31462,33762

Можно одновременно с удалением инстансов удалить БЛОБы, ссылки на которые хранятся в атрибутах этих инстансов. В таком случае после идентификатора инстанса надо поместить амперсанд, кодовое слово blob и через знак равно имя или атрибута, в котором находится удаляемы БЛОБ. Например мы удаляем документ с идентификатором инстанса 1186, две версии этого документа с идентификаторами инстансов 1187 и 1261 и два инстанса типа stored_document с идентификаторами инстансов 1188 и 1262, у которых в атрибутах source хранятся ссылки на БЛОБы которые тоже надо удалить. Запрос будет выглядеть так:

DELETE /rest/1186,1187,1188&blob=source,1261,1262&blob=source

Чтение с сервера Блобов (тел документов)

ВАЖНО! Параметром id указывается идентификатор инстанса!

Формат запроса:

GET /rest/getblob<параметры кодирования>/id=<идентификатор инстанса>&attr=<имя атрибута>/<имя документа для браузера>

·       <параметры кодирования> – опциональный параметр. При значении «enc=b64» тело документа будет предварительно закодировано в base64. При отсутствии параметра тело документа будет передано в оригинальном виде.

·       id=< идентификатор инстанса> – идентификатор инстанса, в атрибуте которого хранится идентификатор блоба, например инстанс типа «apl_stored_document».

·       attr=<имя атрибута> – имя запрашиваемого атрибута, латинские буквы без кавычек, например «source». Вместо имени можно указывать числовой идентификатор атрибута.

·       <имя документа для браузера> представляет собой имя, под которым браузер предложит сохранить скачанный файл. Можно предварительно прочитать его из БД, например, из атрибута инстанса «apl_stored_document» – «file_name».

При вызове извлекает из атрибута <имя атрибута> инстанса с идентификатором <идентификатор инстанса> строчку с идентификатором БЛОБа и возвращает БЛОБ клиенту.

<имя документа для браузера> представляет собой имя, под которым браузер предложит сохранить скачанный файл. Необходимо для корректной работы браузера с файлом и последующей           работы пользователя с файлом; по этому же имени (по расширению) сервер формирует http-заголовок, описывающие тип скачиваемого файла. Если это имя не указано, то файл придет браузеру с нечитаемым именем файла. Если имя файла содержит символы, отличные от символов английского алфавита, то его следует закодировать в процентном виде.
Замечание: в некоторых браузерах, например в Opera, это может не помочь.

Документ возвращается стандартным образом. Если параметры кодирования не заданы, то файл отправляется клиенту в неизмененном виде. Если указан параметр &enc=b64, то файл перед отправкой кодируется в base64. Пример запроса картинки из атрибута source инстанса 5970; картинку перед отправкой необходимо закодировать в base64:

GET /rest&enc=b64/getblob/id=5970&attr=source/logo.jpg

Пример запроса текстового файла из атрибута source инстанса 4526; текст необходимо отправить в оригинальном виде:

GET /rest/getblob/id=4526&attr=source/info.txt

Возвращаемое значение

При успехе запроса данных сервер возвращает код http 200, а в теле ответа – содержимое файла с чистом или закодированном виде.

Если запрос выполнен на неавторизованной сессии (не указан код сессии или сессия с указанным кодом не существует) то сервер возвращает код 401, а в теле ответа – JSON с массивом error_description и строкой с описанием ошибки:

{"error_description": ["Указанный код сессии не действителен. Необходимо пройти авторизацию с помощью команды /rest/connect/"]}

Если неверно указан идентификатор инстанса или имя атрибута, то сервер возвращает код 404 и буфер с описанием ошибки. Если атрибут пустой, или БЛОБа с прочитанным идентификатором нет в БД, то так же возвращается ошибка 404.  Если на указанный инстанс у текущего пользователя нет доступа, то сервер возвращает код 403 и буфер с описанием ошибки.

В случае ошибки чтения из БД возвращается ошибка 500.

Формат запроса:

POST /rest/putblob/id=<идентификатор инстанса>&attr=<имя атрибута>/<имя документа для сохранения в журнал>

·       id=< идентификатор инстанса> – идентификатор инстанса, в атрибут которого будет записан идентификатор блоба, например инстанс типа «apl_stored_document».

·       attr=<имя атрибута> – имя атрибута для хранения блоба, латинские буквы без кавычек, например «source». Вместо имени можно указывать числовой идентификатор атрибута.

·       <имя документа для браузера> имя документа, с которое будет сохранено в журнале создания БЛОБов в случае lite бд, или в поле "name_file" таблицы "temporary_BLOB"

Документ передается в теле http-запроса. Можно использовать тип содержимого «multipart/form-data»; в этом случае в поле данных должен быть представлен один (и только один) файл. Тип содержимого «multipart/byteranges» не поддерживается.  Для остальные типов содержимого подразумевается, что документ передается в теле запроса как единое целое.

При выполнении этой операции происходит следующее:

1)      Файл (тело документа) загружается на сервер/в БД, и ему присваивается новый идентификатор БЛОБа.

2)                  Из инстанса с идентификатором <идентификатор инстанса> атрибута <имя атрибута> читается идентификатор БЛОБа, который был загружен в предыдущей операции загрузки БЛОБа. Если идентификатор в атрибуте есть и БЛОБ с таким идентификатором есть в БД - этот старый БЛОБ удаляется.

3)                  В инстанс с идентификатором <идентификатор инстанса> в атрибут <имя атрибута> записывается идентификатор БЛОБа, загруженного на шаге 1.

4)                  Вызывается операция сохранения данных в БД (commit) или файл Lite БД сохраняется на диск.

Если бы не было шага 2, то после выполнения шага 3 старый БЛОБ в БД становился бы недоступен - так как единственная ссылка на него в атрибуте инстанса была бы перезаписана идентификатором нового БЛОБа.

Возвращаемое значение

При успехе операции сервер возвращает код http 200.

Если запрос выполнен на неавторизованной сессии (не указан код сессии или сессия с указанным кодом не существует) то сервер возвращает код 401, а в теле ответа – JSON с массивом error_description и строкой с описанием ошибки:

{"error_description": ["Указанный код сессии не действителен. Необходимо пройти авторизацию с помощью команды /rest/connect/"]}

Если неверно указан идентификатор инстанса или имя атрибута, то сервер возвращает код 404 и буфер с описанием ошибки. Если на указанный инстанс у текущего пользователя нет доступа, то сервер возвращает код 403 и буфер с описанием ошибки.

В случае ошибки сохранения в БД возвращается ошибка 500.

Тело http-запроса должно содержать данные в формате multipart/form-data. В основном запросе должны присутствовать http заголовок Content-Type со значением "multipart/form-data" и заданным разделителем "boundary=…".

Использование концепции REST для сервера A-YATZK - PSS

Обоснование

Сервер приложений PSS может общаться с обычным клиентом PSS двумя способами: 1) поверх tcp передаются бинарные буферы собственного формата 2) поверх http передаются те же бинарные буферы, завернутые в текстовый вид с помощью base64. В обоих случаях в процессе соединения на сервере создается объект с контекстом клиентской сессии (аутентификация, авторизация, доступ и т.п.); но если в первом случае разрыв сеанса tcp означает удаление объекта и очистку контекста, то во втором случае обработка предусматривает постоянные разрывы соединения, свойственные запросам http. Объект с параметрами сессии живет до явного вызова операции disconnect, или в течении 15 минут после последнего вызова любой функции. Этот же механизм используется для хранения сессии в рамках REST API для обработки одиночных запросов REST поверх http.

REST API PSS предполагает работу на уровне инстансов, их параметров и атрибутов. Соответственно, при обработке REST запросов не требуется знание об объектах высокого уровня (документ, изделие и т.п.), нет необходимости использовать менеджеры и PSS API. Таким образом, компетенции сервера приложений вполне хватает на выполнение запросов REST API, и пропадает необходимость в промежуточном звене в виде дополнительных веб серверов вместе с багами и дополнительными задержками на перекодирование данных между разными представлениями.

Натягивание концепции REST на структуру БД A-YATZK - PSS.

Концепция REST подразумевает, что действия над объектами описываются исключительно методами HTTP; при этом метод HTTP определяет вид операции. Конкретные методы зависят от реализации и могут быть следующими: GET-получить информацию об объекте, POST – изменить объект, PUT – создать объект, DELETE – удалить объект; PATCH – изменить идентификатор объекта и т.п. Однако из практики известно, что методы, отличающиеся от стандартных http-шных GET и POST, могут криво обрабатываться разными программами; кроме того, при сложной структуре БД PSS все многообразие фактически сводится к двум видам операций – чтение данных и их сохранение. В REST API сервера PSS используется только два метода: GET для чтения данных из БД и POST для сохранения измененных данных.

Второй важный момент концепции REST – при запросе данных запрашиваются или все объекты какого-то типа (например, GET /book/ — получить список всех книг), или конкретные объекты с прямой адресацией (GET /book/3/page/35 – получить страницу 35 книги с идентификатором 3). Такой подход хорошо работает только на данных с простой структурой, а у БД PSS сложная структура; данные связаны не только прямыми ссылками, но и связями в виде объектов; из них образуются деревья сложной структуры. Для нормальной работы с такими данными требуется поиск не только по прямым, но и по обратным ссылкам со сложным описанием. Один объект высокого уровня может состоять из нескольких инстансов (см пример с описанием документа в краткой справке по структуре БД); при чтении множества таких составных объектов для ускорения операций возникает необходимость комбинировать несколько связанных подзапросов (частей запросов) в одной операции REST. Предполагается, что в промышленном приложении будут использоваться сложные многосоставные запросы, а простые запросы вида «дай одну инстанс» или «Дай все инстансы одного типа» будут использоваться в основном для тестовых целей.

Третий момент состоит в количестве данных в БД PSS. В боевых БД количество инстансов некоторых типов может достигать десятков и сотен тысяч; выдача их всех браузеру в результате выполнения неоптимального HTTP запроса не приведет ни к чему хорошему. Поэтому результаты всех запросов выдаются порциями; размер порции и индекс начала порции в общем массиве результатов запроса определяется параметрами запроса.

При изменении данных, опять же согласно концепции REST, операции производятся над одним объектом, который описывается в URI запроса (например, POST /book/105/page/26 – заменить страницу 25 книги 105 содержимым из тела запроса, PUT /book/ или POST /book/ - добавить одну книгу). В БД со сложной схемой, в которой объекты высокого уровня представлены несколькими инстансами разных типов, дословное следование этой концепции выльется в кучу мелких запросов при каждом сохранении и приведет к падению производительности вплоть до полной неработоспособности системы. Поэтому в REST API сохранение данных выполняется путем отправки серверу json с описанием одного или нескольких измененных инстансов.

Таким образом, API PSS отходит от строгого следования концепции REST и базируется на следующих моментах:

1)      «Традиционные» REST запросы. Можно запросить у сервера атрибуты одиночной инстанс; можно запросить все инстансы определенного типа (энтити). При запросе всех инстансов определенной энтити можно указать атрибут, по которому будет сортироваться выходной список инстансов.

2)      Запрос данных с помощью составного запроса, аналога запросов CaplLoadData из API С++. В одном составном запросе GET может быть несколько подзапросов, описывающих шаги по получению данных. Параметры подзапросов упаковываются в строку URI стандартными средствами с помощью слешей и символов коньюнкции. Так же вводится аналог запроса NET_FindInstancesWithAttrValues сишного API; его параметры упаковываются в строку URI аналогичным способом. Описание запросов CaplLoadData приведено в документе APL_API.chm

a.       В подзапросах типа «e» и типа FindInstancesWithAttrValues можно задать атрибут, по которому будет сортироваться выходной список инстансов.

b.       В подзапросах типа «a» выходной список сортируется согласно списку инстансов в запросах «a».

c.       В остальных подзапросах инстансы сортируются согласно идентификатору инстанса.

3)      Запрос данных может осуществляться с помощью языка запросов STEP_QL. Символы, недопустимые в строке URI, кодируются в urlencode (процентную кодировку типа %EA%EE%F0%EE%E2%E0 или %u043A%u043E%u0440%u043E%u0432%u0430 для юникода). В запросе этого вида так же можно задать атрибут, по которому будет сортироваться выходной список инстансов, но этот параметр будет работать только для инстансов, у которых есть атрибут с указанным именем. Все прочие инстансы будут располагаться в конце списка и будут отсортированы по идентификатору инстанса.

4)      Для запросов вводятся параметры, позволяющие задать начальный индекс и произвольный размер порции (но максимальный размер порции все равно ограничен). Если в запросе не заданы параметры порций, и количество инстансов больше размера порции по умолчанию, то возвращается порция начиная с нулевого индекса массива результатов и с размером по умолчанию. Т.о. меняя начальный индекс, можно несколькими последовательными запросами получить все результаты запроса. Важно: при выполнении запроса не гарантируется, что с момента предыдущего запроса список результатов не изменился.

5)      Результаты запроса могут включать несколько уровней инстансов, например, в первом подзапросе указаны идентификаторы инстансов папок, а в следующих подзапросах указан атрибут «content» (содержимое папки); в результатах этого запроса верхний уровень будут составлять инстансы типа «apl_folder» (папка), вторым уровнем будут идти инстансы, входящие в указанные папки. Важный момент: на порции делится только список инстансов верхнего уровня; инстансы второго уровня на порции не делятся; поэтому при размере порции 5 и запросе десяти папок с десятью входящими инстансами в каждой результат будет разбит на две порции.

6)      Операция сохранения данных отвязана от конкретных объектов. URI для сохранения выглядит только как «POST /rest/save/» без дополнительных параметров, а все данные об создаваемых, удаляемых и изменяемых инстансах передаются в блоку данных http-запроса одним блоком json. В отладочных целях можно передавать json запросом GET в строке запроса; в таком случае символы, не допустимые в URL, должны кодироваться в urlencode.

7)      Данные об инстансах и их атрибутах передаются в JSON буфере; формат буфера одинаковый для ответов сервера на запросы данных и при сохранении клиентом на сервер измененных данных.

a.       В результатах простых запросов в стиле традиционного REST, запросах типа FindInstancesWithAttrValues и запросах на языке запросов STEP_QL атрибуты типа aplINSTANCE и aplSELECT представляются в упрощенном виде, только параметры инстанса без атрибутов.

b.       При сложном запросе CaplLoadData атрибуты типа aplINSTANCE и aplSELECT, для которых нет явных запросов, так же указываются в упрощенном виде. Значения атрибутов включаются только для тех атрибутов, для которых есть явные указания об их включении в результат.

8)      Запросы REST API PSS должны представлять собой HTTP запросы GET и POST. Строка URI разбивается слешами на группы; каждая группа имеет свое назначение.

a.       Первая группа – указание на используемое API, «rest». В эту же группу добавляются параметры разбиения на порции.

b.       Вторая группа – указывает выполняемое действие («load» - чтение с помощью подзапросов, «query» - запрос на языке запросов, «save» - сохранение и т.п. включая упрощенные запросы).

c.       Последующие группы зависят от выбранного действия. Здесь же задаются параметры сортировки.

9)      Работа с БД PSS/ILS/ПМ ИЛП требует собственной специфической аутентификации и выбора БД для работы. Аутентификация выполняется с помощью запроса POST /rest/connect/<строка описывающая логин, пароль и бд>. Успешный запрос авторизации возвращает в теле ответа идентификатор сеанса, который необходимо передавать во всех последующих запросах в http-заголовке с именем «X-Apl-SessionKey».  После завершения работы желательно завершить сессию запросом POST /disconnect/, но при отсутствии активности сессия автоматически убивается на сервере через некоторое время. Перед аутентификацией можно получить с сервера список имеющихся БД. (Замечание на будущее: предусмотреть передачу параметров коннекта в теле запроса POST с небольшим кодированием, аналогичным прочему кодированию паролей в PSS).

 

Представление инстансов и атрибутов в формате json

Буфер логически разделяется на блок заголовка и на массив инстансов. Блок заголовка состоит из служебных параметров, заданных в виде строк и чисел. Массив инстансов представляет собой массив из объектов, каждый из которых описывает один инстанс и его атрибуты.

Варианты использования данных StepData в формате json

Данные StepData в формате json используются в следующих случаях:

1.       В REST API при передаче с сервера на клиента результатов выполнения запросов.

В БД у всех инстансов гарантировано задан идентификатор инстанса, уникальный в рамках БД, поэтому при передаче с сервера результатов запросов REST заполняется параметр инстанса "id" и не указываются параметры "id_tmp" и "index". Состав передаваемых атрибутов определяется видом операции REST и ее параметрами, т.е. атрибуты должны быть указаны не для всех инстансов. Атрибуты типа «ссылка на инстанс» задаются вложенным объектом json, описывающим инстанс. Если в параметрах запроса REST явно не указано загружать атрибуты инстанса по ссылке, то в объекте jsоn вложенного инстанса включаются только его параметры и не включаются значения его атрибутов.

2.       В REST API при сохранении клиентом на сервер новых инстансов и измененных атрибутов.

При сохранении новых инстансов клиентская часть не может заполнить параметр id (идентификатор инстанса), т.к. этот идентификатор формируется сервером. Идентификатор нового инстанса возвращается клиенту только после успешного сохранения. В то же время, новых инстансов может быть больше одного, и необходимо идентифицировать, какому инстансу был присвоен какой идентификатор. Поэтому клиентская программа должна предварительно занести созданные инстансы в некий массив, и в буфере вы объектах новых инстансов заполнить параметр «index» значениями из этого массива.

  После сохранения сервер возвращает клиенту буфер в формате json, в котором перечислены новые инстансы, у которых заполнены оба параметра: id (идентификатор в БД) и index (индекс в массиве на клиентской части).

Примечание. Если в буфере для какого-то инстанса блок атрибутов задан больше одного раза, и для атрибутов указаны разные значения, то используются последние значения.

 

3.       В CaplStepData при сохранении данных БД формата aplb и формата std в текстовый файл в формате json и при загрузке их из этого формата.

При сохранении данных БД количество инстансов может быть очень большим; при этом в файл должны быть сохранены все значения атрибутов. Кроме того, в БД могут присутствовать закольцовки инстансов (например, инстанс типа «apl_document» ссылается на «apl_document_revision», а «apl_document_revision» ссылается на «apl_document»).  В таких условиях сохранение ссылочных атрибутов как вложенных объектов становится невозможным.

При сохранении в формат json данных из файла std у инстансов может отсутствовать идентификатор, т.к. он создается только при работе сервера приложений; поэтому использование идентификатора для определения ссылочных атрибутов так же недопустимо.

Таким образом, при сохранении всех инстансов из формата std или aplb в формат json:

·       все описания инстансов приведены в массиве «instances» верхнего уровня;

·       каждому инстансу назначается уникальный в рамках файла индекс; этот индекс записывается в атрибут «index»;

·       в ссылочных атрибутах указывается индекс инстанса.

Краткая справка по структуре БД A-YATZK | PSS.

Структура БД определяется словарем БД. Словарь состоит из набора «ентитей» (от английского «entity»); каждая энтити характеризуется идентификатором и именем. И имя, и идентификатор должны быть уникальными в рамках словаря (или комбинации словарей), используемого при работе с БД.

Каждая энтити определяет набор атрибутов. У каждого атрибута есть параметры идентификатор, имя, тип; в зависимости от типа у атрибута могут быть и другие параметры. Идентификатор атрибута должен быть уникальным в рамках словаря или набора словарей; имя атрибута уникально в рамках энтити.

Энтити могут наследоваться друг от друга. Набор атрибутов потомка включает все атрибуты родителя плюс атрибуты собственно потомка. Имена атрибутов потомков не могут переопределять атрибуты родителя.

В рамках REST запросов в основном используется имена энтитей и атрибутов. В некоторых случаях можно указывать идентификаторы; такие случаи оговариваются отдельно.

Собственно данные в БД хранятся в объектах, называемых «инстанс» (от английского «instance»). Каждый инстанс характеризуется несколькими параметрами и содержит значения атрибутов. Параметров всего три: 1) идентификатор инстанса 2) тип инстанса и 3) уровень доступа инстанса.

Идентификатор каждого инстанса уникален и однозначно определяет инстанс в рамках БД.

Тип инстанса – это ссылка на ентити; в рамках REST API энтити обозначается именеем. Тип инстанса (энтити) однозначно определяет набор данных, которые могут храниться в инстансе.

Уровень доступа может иметь 4 значения:

·       полный доступ ("OWN") означает, что текущий пользователь может менять любые значения атрибутов инстанса, назначать на него доступ и удалять его;

·       чтение запись ("RW") означает, что пользователь может менять значения атрибутов;

·       только чтение ("RO") означает, что пользователь может только читать атрибуты;

·       нет доступа ("NA") означает, что пользователь может узнать только факт существования инстанса и его тип, но значения атрибутов прочитать не может.

В рамках БД PSS/ILS/ ПМ ИЛП используются все четыре вида доступа; в специализированных проектах некоторые типы могут не использоваться.

Атрибуты могут иметь следующие типы:

·       aplINTEGER – целое число;

·       aplREAL – дробное число;

·       aplBOOL – булевое значение (true и false);

·       aplLOGICAL – логическое значение (true, false и unknown)

·       aplSTRING – строка;

·       aplENUMERATION – строка, значение которой может принимать только одно значение из заранее заданного списка;

·       aplINSTANCE – ссылка на инстанс конкретного типа; тип инстанса определяется дополнительным параметром атрибута;

·       aplSELECT – ссылка на истанс любого типа;

·       aplAGGR – набор значений одного из выше перечисленных типов; тип значений определяется дополнительным параметром атрибута;

·       aplBINARY, aplBLOB и aplINTEGRATEDBIN – тела документов. В рамках REST API эти типы атрибутов не используются. Для работы с телами документов (БЛОБ, от английского «BLOB» – «binary large object») используются специальные функции.

Пример: в основной схеме apl_pss_a.dict представление документа состоит из следующих типов объектов:

·       document_type – задает тип документов; определяет строчный атрибут «id» (обозначение) и другие атрибуты;

·       document – базовая энтити описания документа; определяет строчные атрибуты «id» (обозначение), «name» (наименование), «description» (описание); атрибут «kind» (тип) является ссылкой на инстанс типа «document_type»; и прочие атрибуты;

·       apl_document (потомок энтити «document») добавляет к атрибутам энтити «document» строчные атрибуты «litera» (литера документа), ссылочный атрибут «active» (активная версия) – ссылка на инстанс типа «apl_document_revision» и другие атрибуты;

·       apl_document_revision – описывает версию документа; имеет строчные атрибуты «revision_id» (версия документа), «description» (описание); атрибут «of_document» является ссылкой на инстанс типа «apl_document»; атрибут «access_form» является ссылкой на инстance типа «apl_stored_document»; прочие атрибуты задают автора версии, литеру, дату, контрольную сумму и прочие параметры версии документа.

·       apl_stored_document – описывает тело документа; строчный атрибут «file_name» хранит имя файла, из которого была загружена в БД версия документа, строчный атрибут «source» хранит идентификатор БЛОБа, в котором находится тело документа.

·       тело документа можно получить с сервер rest-запросом getblob.
